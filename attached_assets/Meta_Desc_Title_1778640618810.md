# Meta_Desc_Title.md

**Purpose:** Handoff document for the Remix Replit. Describes how the parent platform generates the **Meta Title** and **Meta Description** for paired Newsroom articles, including the exact prompts, safety nets, and integration points.

**Audience:** The Remix agent and its owner. Same Next.js / Drizzle / Postgres / OpenAI lineage as the parent — file paths and function names map 1:1 unless noted.

**Important difference from parent:** The parent (Tableicity) is single-tenant and single-brand. **Remix is multi-tenant and multi-brand (5 personas).** Every place this doc says `Tableicity` is a hardcoded brand string in the parent's source — in Remix, those values must be **looked up per tenant** (from a tenants table, an env var, or whatever brand-resolution layer Remix already has). Read the **Multi-Tenant Variances** section before copying any code.

## 0. Multi-Tenant Variances (read this first)

The parent codebase hardcodes the brand name `Tableicity` in seven places. In Remix, each must become a **tenant-resolved variable**. Suggested name in this doc: `BRAND_NAME` (a string resolved from `tenants.brand_name` or equivalent). Apply the same treatment to brand-specific phrasing like "cap table, equity, and 409A guidance" — that's a Tableicity vertical-specific tagline; each Remix tenant has its own.

| # | Where parent hardcodes "Tableicity" | What Remix must do |
|---|---|---|
| 1 | v4 Copywriter system prompt (`lib/newsroom/prompts.ts:574`) — *"You are a city-localization editor for the Tableicity Newsroom (cap-table SaaS)."* | Resolve to `${BRAND_NAME}` and `${BRAND_VERTICAL}` (e.g., "cap-table SaaS", "DTC e-commerce", "regional law firm") per tenant. |
| 2 | v4 Copywriter user prompt metaDescription line (`lib/newsroom/prompts.ts:622`) — *"MUST contain the brand 'Tableicity' AND the city name…"* | Replace with `MUST contain the brand '${BRAND_NAME}' AND the city name '${ctx.cityName}'`. |
| 3 | v3/v4 SEO QC rubric (`lib/newsroom/prompts.ts:559-560`) — *"Meta description missing the brand word 'Tableicity': -10"* | Replace with `Meta description missing the brand word '${BRAND_NAME}': -10`. |
| 4 | `buildMetaDescription()` prefix (`lib/newsroom/pairProcessor.ts:93`) — *"Tableicity in ${cityName}, ${stateCode}: "* | Take `brandName` as a function argument and use it: `${brandName} in ${cityName}, ${stateCode}: `. |
| 5 | `buildMetaDescription()` Tier 3 fallback (`lib/newsroom/pairProcessor.ts:101`) — *"Tableicity insights for founders in…"* + *"Cap table, equity, and 409A guidance…"* | Take `brandName` and `brandTagline` as arguments. The tagline is tenant-specific. |

| 7 | All other prompt files (Researcher, Analyst, Internal Linker) — every system prompt opens with *"You are a … agent for the Tableicity Newsroom (cap-table SaaS)"* | Resolve `${BRAND_NAME}` + `${BRAND_VERTICAL}` per tenant in every system message. |

**You already started this work.** You said you "added an admin with brand injection" in Remix. That's the right pattern. Extend it: brand context flows from the tenant resolver → into a `BrandContext` object → into every `StagePrompt` user/system function → into `buildMetaDescription()` and `buildSuggestedSlug()`.

**Suggested BrandContext shape for Remix:**
```ts
interface BrandContext {
  brandName: string;        // e.g. "Tableicity", "Acme Legal"
  brandSlug: string;        // e.g. "tableicity", "acme-legal" (URL-safe)
  brandVertical: string;    // e.g. "cap-table SaaS", "regional law firm"

```
Pass it through the same way `ctx` (PromptContext) flows today. The simplest wiring: extend `PromptContext` to include a `brand: BrandContext` field, populated by the tenant resolver before `runPipeline()` is called.

## 1. How the parent generates the Meta Title for a paired article

**Headline rule: the parent does not inject the city into an existing title. It generates a new title from scratch with the city locked in as a constraint.**

The Copywriter LLM call lives in `lib/newsroom/openaiGenerator.ts`. It receives the prompt bundle from `lib/newsroom/prompts.ts` (v4 in the Pair flow — hardcoded at `lib/newsroom/pairAgentOrchestrator.ts:104` via `makeOpenAIGenerator("v4")`).

The prompt gives the Copywriter:
- City name + state code as required substitution variables
- The Haylo seed object (cleaned title + body) — this is the "existing title" it draws inspiration from, but it is explicitly instructed *not* to copy it verbatim
- Researcher facts (5-7 named entities specific to the city) and analyst angles as grounding material
- **(Remix addition)** Brand context for the active tenant

The Copywriter must return a title that satisfies:
- **40-90 characters** (enforced by deterministic regex in `checkTitleQuality()` at `lib/newsroom/prompts.ts` near line 683)
- Must contain a **specific number, dollar amount, OR named entity** beyond just the city name
- Must NOT contain any of **29 banned words**: thriving, vibrant, robust, emerging, growing, grows, grow, exciting, innovative, cutting-edge, bustling, dynamic, flourishing, blossoming, booming, burgeoning, ecosystem, scene, landscape, leverage, synergy, revolutionize, unlock, empower, growth, transform, transformative, transforming, game-changing
- No colons, no questions, no em-dashes (em-dash allowed only in dateline)

If the title fails the quality gate, the orchestrator runs `copywriter_retry` (a separate prompt with the failure reasons fed back in). If retry also fails, the result becomes a WARN or FAIL verdict.

**City locking in the title happens at three layers in the parent:**
1. The system prompt names the city: `"Localize this Haylo essay for ${ctx.cityName}, ${ctx.stateCode}."`
2. The dateline field is templated with the city: `"${ctx.cityName.toUpperCase()}, ${ctx.stateCode} — ${dateString}"`
3. The title quality regex does NOT enforce city presence — it's softly enforced via the prompt only.

**Where it lands:** `knowledge_articles.title` column. From there it drives the `<title>` tag in `app/discovery/knowledge/[slug]/page.tsx` `generateMetadata()`.

## 2. How the parent generates the Meta Description for a paired article

**Three-tier defense: LLM first, deterministic body-extractor second (with brand+city prefix), templated sentence as final fallback.**

The parent had a duplicate-content bug for ~6 weeks because the Tier-2 fallback wasn't prefixed. 22 of 67 published articles ended up with byte-for-byte identical meta descriptions — same Haylo essay paired with 22 different cities, all stripped down to the same opening sentences. The fix was to **always prepend `${brandName} in ${cityName}, ${stateCode}: `** in Tier 2. **Build Remix with this fix from day one — do not copy the parent's pre-fix behavior.**

### Tier 1 — LLM
Same Copywriter call as the title. The metaDescription comes back in the same JSON object as the title. Constraints in the v4 prompt:
- **120-280 characters target**
- Must contain the **brand name** AND the **city name** AND at least one named entity from the researcher facts

### Tier 2 — Deterministic body extractor (the safety net)
After the LLM returns, the orchestrator at `lib/newsroom/pairAgentOrchestrator.ts:138-146` checks:

```ts
const metaDescription =
  agentDraft.metaDescription && agentDraft.metaDescription.length >= 40
    ? agentDraft.metaDescription
    : buildMetaDescription(cityName, stateCode, hayloTitle, hayloBodyHtml);
//                          ^ Remix: pass brandName here too
```

If the LLM's metaDescription is shorter than 40 chars, it falls through to `buildMetaDescription()` in `lib/newsroom/pairProcessor.ts:87`. That function:
1. Builds a per-tenant prefix: `${brandName} in ${cityName}, ${stateCode}: `
2. Normalizes the Haylo body HTML to plain text
3. Splits into sentences
4. Greedily concatenates sentences up to `META_DESCRIPTION_TARGET_CHARS - prefix.length`
5. Returns `prefix + sentences`, hard-truncated at `META_DESCRIPTION_HARD_MAX` (300 chars)

This is what makes Tier 2 unique per city — **the prefix carries both brand and city, so even when the body content is identical across cities, the meta description is differentiated.**

### Tier 3 — Templated final fallback
If even the body extractor returns nothing, it uses:
> `"${hayloTitle} — ${brandName} insights for founders in ${cityName}, ${stateCode}."`

Then suffixes with: `" ${brandTagline} for the ${cityName} ${brandAudience}."` if still under 40 chars. Hard-truncates to `META_DESCRIPTION_HARD_MAX`.

### Tier 4 — SEO QC scoring
Whichever tier produced the metaDescription, the SEO QC rubric in `lib/newsroom/prompts.ts:558-560` deducts:
- **-10** if missing OR > 300 chars OR < 80 chars
- **-10** if the brand word is missing
- **-10** if the city name is missing

PASS threshold is 75 (enforced in code at `pairAgentOrchestrator.ts:21`). So an article missing both brand and city in the meta gets a -20 hit and very likely lands in WARN territory (50-74) instead of auto-publishing. Note: the SEO QC LLM prompt itself says "Pass threshold: 80" — this is a known parent-side inconsistency; the orchestrator code (PASS_THRESHOLD = 75) is what actually decides, and the prompt's number is just guidance to the LLM. Remix should pick one number and use it consistently in both prompt text and code.

**Where it lands:** `knowledge_articles.metaDescription` column. From there it drives the `<meta name="description">` tag at `app/discovery/knowledge/[slug]/page.tsx:371`.

## 3. Three things Remix must replicate to get the same behavior

### A. Copy the v4 Copywriter prompt — but parameterize the brand
Located at `lib/newsroom/prompts.ts` in the `v4` PromptBundle, key `copywriter`. Copy the system message, the user-message template, the JSON schema, the character budgets, and the banned-word list. Then **swap every literal `Tableicity` for `${brand.brandName}`** and **swap `cap-table SaaS` for `${brand.brandVertical}`**.

The full v4 Copywriter is reproduced in section 5 below with the parameterization shown inline.

### B. Implement the three-tier metaDescription defense with brand+city prefix
The LLM is unreliable about hitting the 120-280 char target, especially on cities with thin research. The `buildMetaDescription()` body-extractor at `lib/newsroom/pairProcessor.ts:87-107` is the safety net. **Critical: the prefix is what makes Tier 2 unique per city.** Without it, you get the parent's duplicate-content bug.

The full function with parameterization is in section 6.

### C. Use the right prompt version selector per tenant
- Global default in `lib/newsroom/prompts.ts`: `ACTIVE_PROMPT_VERSION = "v2"` (the simple, no-source-grounding version)
- Pair flow override: `pairAgentOrchestrator.ts:104` hardcodes `makeOpenAIGenerator("v4")` (source-grounded, polish-a-Haylo-seed mode)

In Remix, the version selector might also be tenant-scoped (e.g., the legal-firm tenant might use v3 because they don't have Haylo essays). Plan for that — store the active version on the tenant row, not as a single global constant.

## 6. Verbatim buildMetaDescription fallback (with multi-tenant parameterization)

From `lib/newsroom/pairProcessor.ts` lines 62-107:

```ts
function buildMetaDescriptionFromBody(bodyHtml: string, maxChars: number): string | null {
  const normalized = normalizeHayloBody(bodyHtml);
  const text = htmlToPlainText(normalized);
  if (!text) return null;

  const sentences = splitSentences(text);
  if (sentences.length === 0) return null;

  let acc = "";
  for (const s of sentences) {
    const tentative = acc ? `${acc} ${s}` : s;
    if (tentative.length > maxChars) {
      if (!acc) {
        return hardTruncateToPeriod(s, maxChars);
      }
      return acc;
    }
    acc = tentative;
  }
  return acc || null;
}

// PARENT VERSION (single-tenant, brand hardcoded):
export function buildMetaDescription(
  cityName: string,
  stateCode: string,
  hayloTitle: string,
  hayloBodyHtml?: string,
): string {
  const prefix = `Tableicity in ${cityName}, ${stateCode}: `;
  if (hayloBodyHtml) {
    const fromBody = buildMetaDescriptionFromBody(
      hayloBodyHtml,
      META_DESCRIPTION_TARGET_CHARS - prefix.length,
    );
    if (fromBody) return (prefix + fromBody).slice(0, META_DESCRIPTION_HARD_MAX);
  }
  const sentence = `${hayloTitle} — Tableicity insights for founders in ${cityName}, ${stateCode}.`;
  if (sentence.length >= 40) return sentence.slice(0, META_DESCRIPTION_HARD_MAX);
  return `${sentence} Cap table, equity, and 409A guidance for the ${cityName} startup community.`.slice(
    0,
    META_DESCRIPTION_HARD_MAX,
  );
}

// REMIX VERSION (multi-tenant, brand parameterized):
export function buildMetaDescription(
  brand: BrandContext,                      // « ADDED »
  cityName: string,
  stateCode: string,
  hayloTitle: string,
  hayloBodyHtml?: string,
): string {
  const prefix = `${brand.brandName} in ${cityName}, ${stateCode}: `;
  if (hayloBodyHtml) {
    const fromBody = buildMetaDescriptionFromBody(
      hayloBodyHtml,
      META_DESCRIPTION_TARGET_CHARS - prefix.length,
    );
    if (fromBody) return (prefix + fromBody).slice(0, META_DESCRIPTION_HARD_MAX);
  }
  const sentence = `${hayloTitle} — ${brand.brandName} insights for founders in ${cityName}, ${stateCode}.`;
  if (sentence.length >= 40) return sentence.slice(0, META_DESCRIPTION_HARD_MAX);
  return `${sentence} ${brand.brandTagline} for the ${cityName} startup community.`.slice(
    0,
    META_DESCRIPTION_HARD_MAX,
  );
}
```

The constants `META_DESCRIPTION_TARGET_CHARS` and `META_DESCRIPTION_HARD_MAX` are both 300 in the parent. Remix can use the same.

- **What:** Run 10-20 pairs across all 5 tenants. Audit the meta descriptions.
- **Verify:**
  - Tier 2 (deterministic fallback) fires no more than 15-20% of the time. If it fires more often, something is wrong with the LLM call (likely `maxTokens` truncating the response, or the model returning malformed JSON).

  - Brand bleed = zero. Tenant A's articles never contain Tenant B's brand name.
- **Acceptance:** Pass rate at SEO QC threshold ≥ 75 is at least 70% of generated drafts.

## 9. Common pitfalls (lessons from the parent)

1. **Don't lower the title regex thresholds.** The 40-char minimum and the digit-or-entity requirement are what make titles non-garbage. Lowering them = clickbait + duplicate-content risk.

2. **Don't remove the banned-words list.** It's the single biggest contributor to differentiated output. Without it, every article opens with "thriving ecosystem" and reads like 100 identical press releases.

3. **Don't skip the Tier-2 brand+city prefix.** This is the parent's hardest-won lesson. Without the prefix, ~33% of articles will share identical meta descriptions across cities — Google penalizes this as duplicate content. The prefix costs 30-40 chars; it's worth it.

4. **Don't forget the version selector override.** Pair flow MUST use v4. If you only set the global default, the Pair flow silently uses v2 and produces wrong-shape drafts.

5. **Don't conflate the SEO QC "issues" array with hard failures.** PASS verdict means score ≥ 75 (parent's actual threshold; the LLM prompt says 80 but the orchestrator code says 75 — code wins). Issues can still be present and non-empty. The orchestrator publishes anyway. Only score < 50 (FAIL) blocks publication.

6. **(Multi-tenant specific) Don't store brand strings in prompt files.** Put them in the database, on the `tenants` row. Editing prompts to change a brand name across tenants is a recipe for cross-contamination.

7. **(Multi-tenant specific) Test with at least 2 tenants on every gate.** A single-tenant test will pass even if brand context is hardcoded somewhere you forgot. Two-tenant tests catch hardcoded leakage immediately.

## 11. Rules of engagement for the Remix agent

- Plain language, no emojis, no jargon
- Gate-by-gate pacing — finish each gate, summarize, wait for owner's "go" before next gate
- Forward-only — copy the parent's patterns, don't refactor them mid-handoff
- Architect / code review after gates 2, 3, and 5 (each adds new code or new wiring)
- The owner reviews every gate's output before the next gate starts
- If a gate fails, stop and report — don't try to fix it across gate boundaries
- **(Multi-tenant)** Every test must run against at least 2 tenants. A single-tenant pass is not a real pass.

## 12. Appendix: the parent's known historical issues (so Remix doesn't repeat them)

These are bugs the parent shipped, lived with, and eventually fixed. Remix should build them right from day one.

**A. Duplicate meta descriptions across cities (fixed in parent on 2026-05-12).**
Symptom: 22 of 67 published articles had byte-for-byte identical meta descriptions because Tier-2 fallback pulled raw Haylo body sentences with no per-city differentiation. Fix: prepend `${brandName} in ${cityName}, ${stateCode}: ` in `buildMetaDescription`. **Status: live in parent, must be in Remix from day one.**

**B. Brand absent from meta descriptions (fixed in parent on 2026-05-12).**
Symptom: ~34% of published articles had no brand name in the meta description because the v4 Copywriter prompt didn't require it. Fix: prompt now says *"MUST contain the brand 'Tableicity' AND the city name"* and SEO QC deducts -10 for each absence. **Status: live in parent, must be in Remix from day one (with brand parameterized per tenant).**

**C. Not yet fixed in parent — historical articles.** The 23 brand-less + 22 duplicate articles published before the 2026-05-12 fix are still in prod with the old broken meta descriptions. Backfill is a separate decision the parent's owner has not approved yet. Remix should not inherit this debt — build correctly from day one and you have nothing to backfill.
