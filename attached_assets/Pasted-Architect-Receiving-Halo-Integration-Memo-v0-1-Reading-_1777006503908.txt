🔴 The One Redline
Web scrape "in parallel" with Halo consult must become "after Halo consult, conditionally."

The memo says:

"Web scrape still runs in parallel for freshness, but Halo content gets first-class citizenship in the lead pipeline."

Architect's redline: Parallel scrape defeats one of the three benefits you originally sold Halo on — fewer ToS issues, less scrape volume, token savings. If we scrape in parallel every time, we pay the scraping cost regardless of whether Halo answered.

Corrected rule:

Researcher calls Halo first.
If Halo returns ≥1 result with score > 0.75, skip the web scrape for that query. Trust Halo.
If Halo returns zero results OR all results score below 0.75, then fall back to web scrape.
If HALO_ENRICHMENT_ENABLED=false, web scrape runs unconditionally (current Gate 2 behavior).
This preserves the "Halo earns Newsroom savings" thesis. Parallel scraping gives us the quality but not the cost savings. I want both.

One caveat I will accept: For article rewrites (Tier B, the GSC heal loop), parallel scrape IS correct because freshness matters more than cost savings on pages that are already underperforming. I'll revisit this at Gate 5.
