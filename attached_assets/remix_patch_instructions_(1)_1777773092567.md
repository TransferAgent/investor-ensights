# Remix Patch Instructions — Sitemap Freshness + Index Toggle Guard

**Date:** May 3, 2026 (revised)
**Author:** Replit Agent (working with the Tableicity owner)
**Purpose:** Apply two fixes to the Remix that were just shipped to the main Tableicity production app.

**Revision note:** In Fix 2b, `lockedToIndex` is intentionally NOT in the `disabled` prop and is in `aria-disabled` instead. A native HTML disabled button swallows clicks and prevents the "Publish first" toast from ever firing — the user gets silent failure with no feedback. By keeping the button interactive and gating inside the click handler, the toast actually appears. The dim visual (`opacity-60` className) plus the tooltip plus `aria-disabled` for accessibility still telegraph the locked state clearly. If your codebase uses a different control (e.g. a Switch component instead of a button), apply the same principle: keep the control interactive, gate inside the handler, fire the toast.
**Files touched:** 2 (`app/sitemap.ts`, `app/api/admin/knowledge/[id]/route.ts`, `app/admin/knowledge/page.tsx` — three files, two fixes)
**Risk:** Low. No schema changes, no data migrations, no dependency changes. Each change is isolated.
**Deploy required:** Yes, one redeploy after both fixes are in.

## Background — Why These Fixes Exist

We discovered two related bugs that together were causing newly-published Knowledge articles to never reach Google Search Console.

**Bug 1: Sitemap was frozen at build time.**
`/sitemap.xml` was being statically generated by Next.js at deploy time and served from cache forever after. Every database change (new article published, city flipped to indexable, SEO toggle flipped) was correctly persisted, but the sitemap kept replaying the build-time snapshot. Production response header showed `x-nextjs-cache: HIT` as proof.

**Bug 2: The "Index" SEO toggle worked on pending (non-published) articles.**
The Glue & Audit pipeline creates articles with `status = pending` and `robots = "index, follow, ..."` by default. The per-row Index/NoIndex toggle in the Knowledge admin had no guard, so users could flip a pending article's robots to Index even though the article wasn't actually live. This caused the UI to lie about the article's true reachability and added confusion when sitemap inclusion didn't match the toggle state.

The owner's stated vision: **"Published articles show up in sitemap. The toggle means what it means."** These two patches deliver that.

## Fix 1 — Sitemap Freshness (one file, two lines)

### File: `app/sitemap.ts`

**Add these two lines at the top of the file, immediately after the imports and `BASE_URL` constant:**

```ts
export const dynamic = "force-dynamic"
export const revalidate = 0
```

**Full context — before:**

```ts
import type { MetadataRoute } from "next"
import { storage } from "@/lib/storage"

const BASE_URL = process.env.NEXT_PUBLIC_BASE_URL || "https://yourcompany.com"

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  // ...
```

**After:**

```ts
import type { MetadataRoute } from "next"
import { storage } from "@/lib/storage"

const BASE_URL = process.env.NEXT_PUBLIC_BASE_URL || "https://yourcompany.com"

export const dynamic = "force-dynamic"
export const revalidate = 0

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  // ...
```

That is the entire change. No other lines in `app/sitemap.ts` are modified.

### What it does

Tells Next.js App Router: never cache this route. On every request to `/sitemap.xml`, run the function fresh and rebuild the response from live database state.

### Performance impact

Sitemap goes from "served instantly from a static file" to "small Postgres query on every request, roughly 50–200 ms." Sitemaps are requested by Googlebot a few dozen times per day at most. Negligible. If high traffic ever becomes a concern, change `revalidate = 0` to `revalidate = 60` (refresh at most once per minute).

### Risk

None identified. The storage methods called by the sitemap (`getCities`, `getPages`, `getKnowledgeArticles`) are well-tested and unchanged. Rollback is a one-line revert.

## Fix 2 — Index Toggle Guard (two files)

This fix has two layers — a server-side guard (the real protection) and a client-side UX guard (so users get a clear message instead of an opaque API error). Both should be applied.

### File 2a: `app/api/admin/knowledge/[id]/route.ts` — server-side guard

Inside the `PATCH` handler, find the existing line that handles the `robots` field:

```ts
if (body.robots !== undefined) data.robots = body.robots;
```

**Replace it with:**

```ts
if (body.robots !== undefined) {
  const wantsIndex = !String(body.robots || "").toLowerCase().includes("noindex");
  if (wantsIndex) {
    const current = await storage.getKnowledgeArticleById(id);
    if (!current) return NextResponse.json({ error: "Not found" }, { status: 404 });
    if (current.status !== "published") {
      return NextResponse.json(
        { error: "Cannot set robots to Index on a non-published article. Publish the article first, then flip Index." },
        { status: 409 }
      );
    }
  }
  data.robots = body.robots;
}
```

### What it does

When any PATCH request tries to set `robots` to a value that does NOT contain "noindex" (i.e., trying to make the article indexable), the server first re-reads the article from the database. If `status !== 'published'`, the request is rejected with HTTP 409 and a clear message. This catches the bad case even if someone bypasses the UI by hitting the API directly with curl.

### File 2b: `app/admin/knowledge/page.tsx` — client-side UX guard

Find the existing per-row Index/NoIndex toggle button block (around line 1581 in the table cell that renders the SEO badge). It currently looks like this:

```tsx
<TableCell>
  {(() => {
    const indexable = !((a.robots || "").toLowerCase().includes("noindex"))
    return (
      <button
        type="button"
        onClick={() => {
          if (toggleArticleSeoMutation.isPending) return
          const next = !indexable
          if (
            !next &&
            !confirm(
              `Set "${(a.headline || a.title).replace(/<[^>]*>/g, "").slice(0, 60)}" to NoIndex? Search engines will be told to drop this article from results.`
            )
          ) {
            return
          }
          toggleArticleSeoMutation.mutate({ id: a.id, indexable: next })
        }}
        disabled={toggleArticleSeoMutation.isPending}
        data-testid={`button-seo-toggle-article-${a.id}`}
        title={
          indexable
            ? "Indexable — click to switch to NoIndex"
            : "NoIndex — click to switch to Index"
        }
      >
        {indexable ? (
          <Badge variant="outline" className="border-green-500/50 text-green-600 dark:text-green-400 hover-elevate cursor-pointer">
            Index
          </Badge>
        ) : (
          <Badge variant="secondary" className="text-muted-foreground hover-elevate cursor-pointer">
            NoIndex
          </Badge>
        )}
      </button>
    )
  })()}
</TableCell>
```

**Replace the inner IIFE with this version (adds `isPublished`, `lockedToIndex`, the toast, the disabled state, and the dimmed visual):**

```tsx
<TableCell>
  {(() => {
    const indexable = !((a.robots || "").toLowerCase().includes("noindex"))
    const isPublished = a.status === "published"
    const lockedToIndex = !isPublished && !indexable
    return (
      <button
        type="button"
        onClick={() => {
          if (toggleArticleSeoMutation.isPending) return
          if (lockedToIndex) {
            toast({
              title: "Publish first",
              description: "Index can only be turned on for published articles. Publish this article, then flip Index.",
              variant: "destructive",
            })
            return
          }
          const next = !indexable
          if (
            !next &&
            !confirm(
              `Set "${(a.headline || a.title).replace(/<[^>]*>/g, "").slice(0, 60)}" to NoIndex? Search engines will be told to drop this article from results.`
            )
          ) {
            return
          }
          toggleArticleSeoMutation.mutate({ id: a.id, indexable: next })
        }}
        disabled={toggleArticleSeoMutation.isPending}
        aria-disabled={lockedToIndex}
        data-testid={`button-seo-toggle-article-${a.id}`}
        title={
          lockedToIndex
            ? "Publish this article before you can switch it to Index"
            : indexable
            ? "Indexable — click to switch to NoIndex"
            : "NoIndex — click to switch to Index"
        }
        className={lockedToIndex ? "cursor-not-allowed opacity-60" : undefined}
      >
        {indexable ? (
          <Badge variant="outline" className="border-green-500/50 text-green-600 dark:text-green-400 hover-elevate cursor-pointer">
            Index
          </Badge>
        ) : (
          <Badge variant="secondary" className="text-muted-foreground hover-elevate cursor-pointer">
            NoIndex
          </Badge>
        )}
      </button>
    )
  })()}
</TableCell>
```

### Behavior matrix after both guards are in place

| Article state | Click "Index" badge | Click "NoIndex" badge |
|---|---|---|
| pending, currently NoIndex | **Blocked** with red toast "Publish first" | n/a (already NoIndex) |
| pending, currently Index (legacy bad state) | n/a (already Index) | **Allowed** — lets you clean up |
| published, currently NoIndex | Allowed | n/a |
| published, currently Index | n/a | Allowed |
