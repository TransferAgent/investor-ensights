# City-Aware Tableicity Backlink in Published Articles

## Context

Every published Knowledge article (press release) used to have its first in-body "Tableicity" mention auto-linkified to the homepage `https://www.tableicity.com`. Since each article is paired to a specific city, we wanted that link to point at the city's location page instead â€” for example, an article published for Abilene, TX should link to `https://www.tableicity.com/locations/abilene-tx`.

## Decision Summary

- **Single-file change.** All article rendering for `/discovery/knowledge/[slug]` happens server-side in one component, so the rewrite happens at render time only.
- **No database change, no backfill, no LLM cost.** Article body HTML in `knowledge_articles.body_html` already stores plain text "Tableicity" mentions (no anchor tags). The link is created at view time by a small regex transformer. Editing that transformer instantly applies to all 67+ existing published articles on next page load.
- **Forward- and backward-compatible.** Articles without a `citySlug` (rare legacy direct-from-city generations) gracefully fall back to the homepage link.
- **Anchor text unchanged.** The visible word stays "Tableicity" â€” clean prose, no awkward "Tableicity in Abilene, TX" rewording.

## Files Touched

Only one file:

- `app/discovery/knowledge/[slug]/page.tsx`

No schema migration, no API change, no admin UI change.

## Step-by-Step Implementation

### 1. Located the auto-link function

Searched `app/discovery/knowledge/[slug]/page.tsx` and found a function `highlightBrandInBody` (lines ~7â€“28) that:

- Splits the article HTML into tokens.
- Tracks whether the parser is currently inside an `<a>` tag (to skip already-linked text).
- Replaces the **first** plain-text occurrence of `\bTableicity\b` with an anchor pointing to `https://www.tableicity.com`. All subsequent occurrences are left as plain text to avoid spammy repeated links.

This function was called by `transformBodyForRender(html, subheadline)` (line ~325), which in turn was invoked inside the body's `dangerouslySetInnerHTML` block (line ~533).

### 2. Confirmed `citySlug` is available at the call site

The `article` object passed into the page already carries `article.citySlug` (column on `knowledge_articles`). No extra DB fetch needed.

### 3. Made three small, batched edits

**Edit A â€” `highlightBrandInBody` signature and href construction:**

```ts
function highlightBrandInBody(html: string, citySlug?: string | null): string {
  let firstReplaced = false;
  let inAnchor = false;
  const href = citySlug
    ? `${BASE_URL}/locations/${citySlug}`
    : "https://www.tableicity.com";
  const tokens = html.split(/(<[^>]*>)/);
  return tokens
    .map((token) => {
      if (token.startsWith("<")) {
        if (/^<a[\s>]/i.test(token)) inAnchor = true;
        else if (/^<\/a>/i.test(token)) inAnchor = false;
        return token;
      }
      if (inAnchor) return token;
      return token.replace(/\bTableicity\b/g, () => {
        if (!firstReplaced) {
          firstReplaced = true;
          return `<a href="${href}" class="text-blue-400 hover:underline">Tableicity</a>`;
        }
        return "Tableicity";
      });
    })
    .join("");
}
```

Key changes:
- New optional second parameter `citySlug?: string | null`.
- New `href` constant â€” built from `BASE_URL + "/locations/" + citySlug` if a slug is provided, otherwise the homepage URL (preserves prior behavior for legacy/null-slug articles).
- The anchor template literal now uses `${href}` instead of the hardcoded homepage URL.

`BASE_URL` is already declared at the top of the file:
```ts
const BASE_URL = process.env.NEXT_PUBLIC_BASE_URL || "https://www.tableicity.com";
```

**Edit B â€” `transformBodyForRender` passes `citySlug` through:**

```ts
function transformBodyForRender(
  html: string,
  subheadline?: string | null,
  citySlug?: string | null,
): string {
  return markFirstAnswerBlock(
    ensureAnswerBlock(
      highlightBrandInBody(
        rechunkParagraphs(removeDuplicateLeadParagraph(html, subheadline)),
        citySlug,
      ),
    ),
  );
}
```

Added the third parameter and threaded it down to `highlightBrandInBody`.

**Edit C â€” Call site supplies `article.citySlug`:**

In the JSX body render block (~line 543):

```tsx
dangerouslySetInnerHTML={{
  __html: transformBodyForRender(
    article.bodyHtml,
    article.metaDescription || article.subheadline,
    article.citySlug,
  ),
}}
```

### 4. Verified live

After Fast Refresh picked up the changes, ran a curl probe against a real published article:

```bash
curl -s http://localhost:5000/discovery/knowledge/tableicity-austin-cap-table \
  | grep -oE '<a[^>]*href="[^"]*"[^>]*>Tableicity</a>'
```

Output:
```
<a href="https://www.tableicity.com/locations/austin-tx" class="text-blue-400 hover:underline">Tableicity</a>
```

Confirmed: the first "Tableicity" mention now anchors to the city page (`/locations/austin-tx`) instead of the homepage.

## What Did NOT Change

- `knowledge_articles.body_html` (DB) â€” untouched. Plain "Tableicity" text remains.
- The 5-agent generation pipeline (`pairAgentOrchestrator`, `pipelineWorker`, prompts) â€” untouched.
- The "About Tableicity" boilerplate paragraph at the very bottom of articles â€” that uses `article.boilerplateHtml` and is rendered separately. If you ever want its trailing `https://www.tableicity.com` link to also retarget per-city, that's a separate small edit (rewrite the `<a href>` in `boilerplate_html` either at publish time via a helper or at render time via a similar regex). Not done in this gate.
- Any anchor that was already inside an `<a>` tag in the source HTML â€” preserved as-is (the `inAnchor` guard skips them). This means future internal-linker-injected anchors are safe.
- HTML sanitizer (`lib/newsroom/htmlSanitizer.ts`) â€” untouched.

## Edge Cases Handled

- **`article.citySlug` is null** â†’ falls back to `https://www.tableicity.com` (original behavior preserved).
- **Multiple "Tableicity" mentions in body** â†’ only the first is linkified; the rest stay as plain text. This was already the existing behavior; we did not change it.
- **Existing anchors around the word "Tableicity"** â†’ skipped via the `inAnchor` flag; not double-wrapped.
- **`BASE_URL` env var override** â†’ respected. If `NEXT_PUBLIC_BASE_URL` is set (e.g. for staging or preview), the city link uses that base instead of hardcoded `tableicity.com`.

## How to Apply This to a Remix

In your remix, find the equivalent of `app/discovery/knowledge/[slug]/page.tsx`. Look for a function that auto-linkifies the brand name in the article body â€” search for `Tableicity` (or your remix's brand name) inside an anchor template literal. Apply the same three edits:

1. Add an optional `citySlug?: string | null` parameter to the linkifier function and build the href conditionally from `${BASE_URL}/locations/${citySlug}`.
2. Thread `citySlug` through any wrapper function that calls the linkifier.
3. At the JSX call site that renders the article body, pass `article.citySlug` as the new argument.

That's the entire change. No DB migration, no API change, no backfill.
