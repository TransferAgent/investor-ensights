## What This Is

Investor Ensights is the umbrella publisher for a portfolio of vertical financial-news brands. Each brand (Tableicity, Veltroy, Haylo, Texitie, Payrol) operates as a tenant on the same platform, sharing one editorial standard, one author masthead, and one infrastructure — while maintaining its own content, its own city catalogue, and its own audience.

The platform exists to solve one problem at scale: **how to publish institutionally rigorous, locally specific financial coverage across thousands of US markets without sacrificing editorial quality or breaking SEO.**

Today the platform manages:
- **5 tenants** (sister brands) publishing through one domain
- **1,688 city pages** spanning all US states
- **90 published knowledge articles** (press releases) with full E-E-A-T author attribution
- A **5-agent AI newsroom pipeline** that drafts, fact-checks, and ships content under human editorial review

## Architecture

### Multi-Tenant: Schema-per-Tenant

Each tenant lives in its own PostgreSQL schema (`tenant_<slug>`). The `public` schema holds platform-level tables (users, tenants, tenant membership, global URL registries). Tenant context is **session-derived** — every user belongs to exactly one tenant via `tenant_members`. No tenant-picker UI, no subdomain routing, no path prefix on admin routes.

**Key invariant:** the persona slug *is* the schema name *is* the article URL prefix. One string, three roles:
- `tableicity` → `tenant_tableicity` schema → `/discovery/knowledge/tableicity-…`

### Global vs Per-Tenant Tables

| Scope | Tables |
|---|---|
| **Global (`public`)** | `users`, `tenants`, `tenant_members`, `email_verifications`, `city_slug_registry`, `session` |
| **Per-Tenant (`tenant_<slug>`)** | `knowledge_articles`, `city_locations`, `haylo_articles`, `newsroom_*` (9 tables), `content_templates`, `custom_pages`, `admin_audit_log`, and more |

City URL slugs are enforced **globally unique** via `public.city_slug_registry` so two tenants can never publish conflicting URLs. Article slugs are globally unique by construction (persona prefix).

### The Newsroom Pipeline

Content is generated by a **5-agent model** running in parallel where possible:

1. **Researcher** — pulls grounding sources from `city_research_sources`
2. **Data Analyst** — extracts and validates numerical claims
3. **Local Copywriter** — drafts the narrative in the persona's brand voice
4. **SEO QC** — enforces title/description contracts (Google SERP fit)
5. **Internal Linker** — wires backlinks to persona hub + city pages

Gating rules ensure:
- Every article body mentions the persona brand at least once (word-boundary check before the draft is composed)
- Meta titles target 55 chars (max 65), exclude the brand, must contain the city
- Meta descriptions target 150 chars (range 100-200), brand mentioned 1-2x but never in the first 40 chars
- Cities require ≥1 enabled research source to be eligible for the auto-scheduler

A two-shot **meta naturalizer** (`metaNaturalizer.ts`) polishes formula-generated titles/descriptions with a small LLM call (~$0.0006/article worst case), with formula fallback if both LLM attempts fail.

### E-E-A-T Author Attribution

Every article, every tenant, every city is attributed to a single named masthead author with full `Person` and `Organization` JSON-LD schema, contact details, and LinkedIn `sameAs`. The author byline and bio render decoupled from the database `author_name` column — the DB field is data-hygiene, the display is driven by a single source of truth in `lib/author-config.ts`. This guarantees identity consistency across 90+ articles without per-row drift.

### Persona Wizard (Tenant Onboarding)

Conductor-only `/admin/personas` flow that creates a new tenant atomically (single Postgres transaction: insert into `public.tenants` + provision `tenant_<slug>` schema + seed `tenant_members`). Brand voice is **derived** from a pasted Haylo essay by an LLM call rather than hand-typed, so new tenants ship with a coherent editorial voice from day one. Adding a tenant no longer requires SQL access.

## Platform Capabilities

### Public Site
- **City Landing Pages** (`/locations/[slug]`) — city-specific content, slideshows, geo-detected contact info, embedded maps
- **Knowledge Articles** (`/discovery/knowledge/[slug]`) — SEO-optimized press releases with full JSON-LD, OpenGraph, dynamic sitemaps, robots.txt support
- **Locations Grid** (`/locations`) — search, filter, and geo-detection across all 1,688 cities
- **Custom Pages** (`/[slug]`) — slide-based content builder for marketing pages (About, Terms, Privacy, etc.)
- **About & Editorial Team** (`/about`) — publisher mission, masthead, Person + Organization schema
- **Dynamic Sitemap** (`/sitemap.xml`) — auto-generated, public-cacheable, no cookies

### Admin Dashboard
- Full CRUD for cities, content templates, custom pages, knowledge articles
- Bulk operations (CSV upload for city batches and Haylo libraries)
- User management with last-admin delete protection
- Audit log per tenant (`tenant_<slug>.admin_audit_log`)
- Newsroom Auto-Scheduler with configurable drip publishing
- Haylo library: essay ingestion, duplicate detection, sandboxed previews
- Persona Wizard (Conductor only) for adding new tenants without SQL

### SEO
- JSON-LD `NewsArticle` schema on every article (author Person, publisher Organization with ImageObject logo, mainEntityOfPage, dateModified)
- JSON-LD `Person` + `Organization` schema on `/about`
- OpenGraph + Twitter Card metadata
- Sitemap-as-canary monitoring (live URL count tracked as a regression alarm)
- Forward-only deletes policy — no retroactive slug removal without sign-off

### Security
- Session-based auth with bcrypt password hashing
- Two-phase login: password → 6-digit MFA code
- Rate limiting on admin user-management endpoints
- Atomic last-admin delete prevention
- Input sanitization on all user-generated HTML
- Tenant isolation enforced at the database `search_path` level

## Getting Started

### Prerequisites
- Node.js 20+
- PostgreSQL 14+ (or a hosted equivalent — Neon, Supabase, Replit Database)

### Install
```bash
npm install
```

### Generate Drizzle types
```bash
npm run db:generate
```

### Push schema to dev DB
```bash
npm run db:push
```

### Run dev server
```bash
npm run dev
```

The dev server runs both the Next.js app and its API routes on a single port.

## Operating the Platform

### Daily / weekly
| Command | Purpose |
|---|---|
| `npm run dev` | Local dev server |
| `npm run build` | Production build |
| `npm run typecheck` | TypeScript validation |

### Schema management
| Command | Purpose |
|---|---|
| `npm run db:push` | Push schema to **dev** DB (public schema only) |
| `node scripts/sync-tenant-schemas.mjs` | Propagate additive schema changes to every `tenant_<slug>` schema (dev). Add `--prod` for production. Add `--dry-run` to preview. |
| `bash scripts/push-schema-to-prod.sh` | Push schema to **prod** DB and auto-sync tenant schemas |

> ⚠️ **Schema-per-tenant drift:** `drizzle-kit push` only touches the `public` schema. After ANY schema change, also run the tenant-sync script or each `tenant_<slug>` will silently fall out of sync with the type definitions.

### Data sync
| Command | Purpose |
|---|---|
| `npx tsx scripts/sync-dev-to-prod.ts --confirm` | Sync dev data to prod (omit `--confirm` for dry-run) |

### One-shot backfills
| Command | Purpose |
|---|---|
| `npx tsx scripts/backfill-tableicity-meta.ts [--prod] [--confirm] [--force]` | Re-stamp SEO meta titles/descriptions through the current naturalizer contract. Idempotent by default — skips rows where `meta_locked_at IS NOT NULL`. `--force` only for approved re-locks. |
| `npx tsx scripts/eeat-author-and-contact-rollup.ts --prod --confirm` | Stamp masthead author + contact details across all articles, tenants, and city pages in one atomic transaction. |

## Deployment

The platform deploys to Replit Deployments. After any schema change:

1. Push schema to prod: `bash scripts/push-schema-to-prod.sh`
2. Sync data dev → prod (if needed): `npx tsx scripts/sync-dev-to-prod.ts --confirm`
3. Deploy via the Replit Deployments UI
4. Resubmit the sitemap in Google Search Console

### Sitemap-as-canary

Live `investorensights.com/sitemap.xml` should grow monotonically. The current baseline is **105 URLs** (89 published articles + persona hubs + orphan + locations grid + `/about` + static). A SHRINK below baseline is the real alarm — it means the forward-only deletes policy was violated, or a noindex sweeper got reintroduced.

## Editorial Standard

Every article published through any tenant under the Investor Ensights masthead must:
- Cite an underlying public data source
- Name the jurisdiction it covers
- Explain what the numbers mean for the reader
- Carry the standard byline (author Person + publisher Organization)
- Pass the meta-title and meta-description gates (SERP fit, brand-out-of-title, 80/20 content lead)
- Mention the persona brand at least once in the body (word-boundary check)

These are enforced at the pipeline level, not by editorial review alone.
