# replit.md

## Overview

This project is a **City Landing Page Manager**, a full-stack web application designed for programmatic SEO. It allows the creation and management of location-based landing pages across various US cities using reusable content templates with dynamic placeholder substitution. The application features a public-facing website for browsing city locations and templated content, alongside an admin dashboard for comprehensive management of cities, content templates, and their assignments.

## User Preferences

Preferred communication style: Simple, everyday language.

## System Architecture

The application is built with **Next.js 16** (App Router), **React 18** (TypeScript), **Tailwind CSS 3** for styling, and **shadcn/ui** for UI components. Data fetching on the admin side is handled by **TanStack React Query**.

**Key Features:**

*   **Public-facing Pages:**
    *   Dynamic city landing pages (`/locations/[slug]`) generated from templates, featuring city-specific content, slideshows, contact info, and maps.
    *   A locations grid (`/locations`) with search, filter, and geo-detection capabilities.
    *   Custom pages (`/[slug]`) created via a Page Builder.
    *   Public press release/knowledge articles (`/discovery/knowledge/[slug]`) with full SEO support (JSON-LD, OpenGraph, dynamic sitemaps, robots.txt).
*   **Admin Dashboard:**
    *   Login and user authentication using JWT and httpOnly cookies.
    *   CRUD operations for cities, content templates, custom pages, and knowledge articles.
    *   Bulk operations for cities (CSV import, publish/unpublish, template assignment).
    *   A Page Builder for creating and managing custom pages with configurable content slides.
    *   A "Knowledge" section for managing press releases, including content studio for generation, template management, analytics, and coverage tracking.
    *   A Newsroom pipeline that has pivoted twice. Originally an "AI writer". Then a slim 4-stage Pair → Glue → Audit → Schedule flow (Glue = `pressReleaseComposer.ts` wrapper, Audit = single gpt-4o-mini grader in `auditor.ts`). **Now** the live (non-dry-run) Pair flow uses the **full 5-agent pipeline** in v4 polish-mode: Researcher → Data Analyst → Local Copywriter → SEO QC → Internal Linker, orchestrated by `lib/newsroom/pairAgentOrchestrator.ts` (~$0.0024/pair, ~30s, ~13k tokens against gpt-4o-mini). The orchestrator: (1) calls `composePressRelease` for a canonical dateline + Glue seed (kept in `PairResult.composed` for type compatibility), (2) calls `ensureCitySources(citySlug)` from `cityResearchAutoSeeder.ts` which auto-derives Wikipedia / city.gov / Chamber / Crunchbase URLs from `city_locations` metadata, SSRF-safely probes one URL per family with a `MIN_CONTENT_BYTES=800` floor, and inserts productive ones via `INSERT…ON CONFLICT DO NOTHING` into `city_research_sources` (junk URLs → fallback → skip → never stall — boston-ma went 0→3 sources in smoke test), (3) calls `runPipeline` with `promptVersion="v4"`, `skipReviewQueue=true`, and `hayloSeed={title, bodyHtml, topicSlug}` so the orchestrator owns the saving destination instead of `runPipeline` always inserting into `newsroom_review_queue`, (4) maps `qcScore` to verdict (PASS≥75, WARN 50–74, FAIL<50) and reshapes `qcIssues` strings into categorized `AuditorIssue` objects so the result has the same `PairResult` shape as the legacy `processPair`. The `enqueue-pairs` route then keeps its existing PASS→`knowledge_articles` / WARN→`newsroom_review_queue` / FAIL→`knowledge_generation_log` saving logic unchanged. Dry-run still falls through to the cheap mock `processPair` (no LLM cost). The v4 prompt bundle (`lib/newsroom/prompts.ts`) inherits all v3 source-grounded stages (Researcher/Analyst/QC) and only specializes the Copywriter into "lede-only" mode: the Copywriter sees the Haylo seed via `extras.hayloSeed`, emits a 1–2 paragraph city-specific opening lede + fresh title/headline/dateline (no Tableicity CTA, no `<h2>`), and `pipelineWorker.ts` then stitches `normalizeHayloBody(hayloSeed.bodyHtml)` from `lib/newsroom/hayloBodyNormalizer.ts` (strips first `<h1>`, leading dateline-like `<p>`, all `<strong>`/`<b>` tags) AFTER the lede before SEO QC scores the full draft. v4 is in `SOURCE_GROUNDED_VERSIONS`, so all v3 source-grounding gates (Researcher must cite `sourceUrl`, QC deducts -20 for ungrounded facts) still apply. The legacy `auditor.ts` slim grader, `pressReleaseComposer.ts`, and `hayloStitcher.ts` remain as the dry-run path and as the canonical dateline source. The v3 Analyst's `localVibe` ("insufficient commercial signal..." sentinel) and door-hanger publish gates are not exercised in the new Pair-agent flow because the Copywriter writes from researcher facts directly rather than receiving a pre-extracted vibe string. Each placement = one `knowledge_articles` row keyed by city slug.
    *   Data store for managing file uploads.
    *   **Newsroom Auto-Scheduler** (drip publishing): A configurable scheduler that auto-runs the 5-agent Pair pipeline at a steady cadence. Singleton config in `newsroom_scheduler_config` (enabled flag, `pairings_per_day` default 5, `daily_budget_usd` default $1.00, `picker_strategy` "balanced"|"newest_first"|"random"). Picker (`lib/newsroom/schedulerPicker.ts`) selects the next `(haylo_articles × city_locations)` pair, excluding any pair already present in `knowledge_articles` or pending in `newsroom_review_queue` (matched via `draft_payload->>'hayloArticleId'`). Runner (`lib/newsroom/schedulerRunner.ts` — `runSchedulerTick`) gates on enabled / quota / budget / OpenAI key / in-process tickInFlight mutex / eligible pair, then calls `runPairAgentPipeline`, saves PASS→`knowledge_articles` (status=pending) + bumps `haylo_articles.placement_count` atomically in one transaction, WARN→`newsroom_review_queue`, FAIL→`knowledge_generation_log`, and writes a row to `newsroom_scheduler_runs` (history). A top-level catch converts any post-pipeline persistence crash into an `outcome="error"` history row. The in-process mutex is sufficient for single-instance Replit deployments; multi-instance would need a Postgres advisory lock instead. Cron endpoint at `/api/cron/newsroom-scheduler` accepts `CRON_SECRET` via `x-cron-secret` header or `Authorization: Bearer …` (header-only — no query-param secret to avoid leaking via logs/referers). Admin endpoints under `/api/admin/newsroom/scheduler/{config,runs,tick-now}` are session-authed. UI lives in `app/admin/knowledge/scheduler-card.tsx` (mounted in the Content Studio tab of `/admin/knowledge`): toggle, pairings-per-day input, budget input, picker-strategy select, today's stats (pairings/passes/warns/fails/spend/tokens), eligible-pair count, last-20-ticks list (15s refetch), and "Run a tick now" button. Run history table `newsroom_scheduler_runs` records every tick with outcome, city, haylo, verdict, flow score, knowledge/review IDs, cost, tokens, duration, notes.
*   **SEO Enhancements:** Server-Side Rendering (SSR), dynamic sitemap and robots.txt, canonical URLs, OpenGraph tags, and JSON-LD structured data are implemented across public pages.
*   **Authentication & Security:** JWT-based authentication with `scrypt` password hashing, httpOnly cookies, rate limiting for login attempts, input sanitization against XSS, SQL injection protection via Drizzle ORM, and comprehensive admin audit logging.
*   **Geocoding:** Integration with OpenCage API for automatic latitude/longitude population for cities.

**Data Storage:**
The application uses **PostgreSQL** as its database, accessed via **Drizzle ORM** and `drizzle-zod` for schema validation. Key database tables include `city_locations`, `content_templates`, `city_content_assignments`, `admin_users`, `admin_audit_log`, `custom_pages`, `page_slides`, `knowledge_articles`, `knowledge_article_versions`, `knowledge_generation_log`, `knowledge_campaigns`, `data_store_files`, `knowledge_templates`, `city_research_sources` (per-city seed URLs for the source-grounded Newsroom Researcher), and `haylo_articles` (the Haylo Library — production-ready essays from Haylo Lab keyed by sha256 `content_hash` unique index, with `placement_count` tracking how many cities have paired with each article).

**Haylo Library (admin):** `/admin/haylo` ("Haylo Library" sidebar entry between Pages and Knowledge) provides paste-in and file-drop ingestion for Haylo Lab essays. The paste form stores any HTML byte-for-byte (no sanitization — Haylo content is trusted). The "Scan Inbox" button reads `.html` files from the `haylo-inbox/` folder, parses an optional `<!-- topic: X -->` comment for the topic slug (falling back to filename), auto-detects title from `<h1>` → `<title>` → first `<p>` → filename, and ingests with `source="inbox-import"`. Duplicates are detected by sha256 of body HTML and skipped (also enforced by a unique DB index). Articles have status `ready`/`draft`/`retired`; only `ready` will be eligible for Pair-stage selection. Delete is blocked at the API and UI level whenever `placement_count > 0` to prevent removing an article that's already in published press releases. Preview uses a sandboxed `<iframe sandbox="" srcdoc>` so untrusted HTML cannot script the admin shell. All mutations write to `admin_audit_log`. API: `GET/POST /api/admin/haylo-articles`, `GET/PATCH/DELETE /api/admin/haylo-articles/[id]`, `POST /api/admin/haylo-articles/scan-inbox`. Storage primitives in `lib/haylo/ingest.ts` (`hashHaylo`, `parseHayloFile`, `buildInsertFromPaste`, `ensureUniqueSlug`).

## External Dependencies

*   **PostgreSQL**: Primary database for all application data.
*   **OpenCage API**: Used for geocoding services to auto-fill city latitude and longitude.
*   **OpenAI API**: Utilized for the Newsroom pipeline for AI-driven content generation, specifically with the `gpt-4o-mini` model.
